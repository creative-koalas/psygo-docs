# PsyGo用户指南

总体指南：把PsyGo当成一个刚入职啥都不懂但能力很强的实习生，把PsyGo App当做跟这个实习生沟通用的微信。
把PsyGo当一个人类实习生，而不是Claude Code，WorkBuddy之类的工具，更不是豆包之类的聊天机器人。

## PsyGo可以很强，但不能读心

刚入职的PsyGo是一张白纸，能做很多工作：

- Word文档编写
- 表格分析
- 调研报告
- PPT制作
- 全栈软件开发
- 技术研究
- ...

但在没有上下文的情况下，PsyGo只能按照**他/她对任务的理解**工作，
无法按照**你的理解**工作。

如果想要PsyGo和你的具体工作对齐，你需要向PsyGo提供足够的信息。
例如，这样要求PsyGo大概率得不到你想要的：

>*帮我做个OPC平台*

这样要求PsyGo能更好地让他理解任务：

>*帮我做个湖北省武汉市江汉区OPC平台的介绍网站，做之前建议到网上查找相关政策，保证信息准确*

好消息是，**PsyGo具有学习和记忆能力**。
对于同一个员工，你不需要每次都重复同样的指令，而是可以先教会他怎么干活：

>*记好了，我们部门做行业调研要先到网上查资料，交付方式是PDF，
用latex做，参考信息来源要标清楚，注意时效性，尽量找两年以内的信息*

然后：

>*给我出个AI游戏行业的调研报告，按我之前教你的来*

## PsyGo不是工具，不要在App上等结果

PsyGo无法在你发消息后立刻回复，因为PsyGo不是聊天机器人，而是能干活的工人。

正确的交互方式是：**下命令之后，就关掉客户端窗口，然后该干啥干啥**。
PsyGo把工作做完了，或者需要你提供消息了，自然会来找你。
就像一个人一样。

可以把PsyGo当作微信上的一个联系人。正在干活的人是很难秒回的，但是他/她干完了自然会联系你。

**PsyGo的设计理念就不是让你一直呆在App上**，
而是反过来，让AI尽量自主工作，尽可能省心，把更多的时间还给用户。

## 把PsyGo当个远程工作的实习生即可

不要假定PsyGo不能做什么，不需要担心“我这么做了他会不会坏掉”，把PsyGo当个人，
把PsyGo App当这个人的微信即可。

你可以给PsyGo发任何东西，PsyGo会想办法读取和处理它们。例如：

- Word文档
- Excel表格
- PPT
- PDF
- 网页链接
- 压缩包
- ...

同理，PsyGo也可能给你发：

- Word文档
- Excel表格
- PPT
- PDF
- 网页链接
- 压缩包
- ...

## PsyGo用不了你的电脑，但PsyGo有自己的电脑

与OpenClaw不同，PsyGo并不运行在你的电脑上；
每个PsyGo都有一台自己的电脑。

PsyGo只能看到你告诉他的东西或者发给他的文件，或通过其他方式（如gitee）授权他使用的信息，
无法看到你本地的电脑。

**你需要把工作需要的信息和文件发给PsyGo，或通过其他方式授权给他，他才可以使用这些东西。
只要你不发，PsyGo看不到你电脑/手机上的任何东西。**

## 不要把PsyGo当聊天机器人，不要像用豆包一样用PsyGo

**PsyGo不是聊天机器人，也不是AI工具，而是AI工人。**

**不要用豆包/ChatGPT/Claude Code的方式使用PsyGo。**

**PsyGo App只是管理员后台+微信，不是PsyGo的全部**，
就像一个人远不只是他/她的微信账号。

实际的交付应该是通过发文件，git等等，而不是在PsyGo App上发消息，
**就像没有人会把代码文件复制粘贴到微信里**。

**不要让PsyGo：**

- 搞复杂的调研，然后在聊天框里汇报
- 在你的电脑上操作，比如给你本地配环境，在你本地写代码
- 像Claude Code一样在你本地调代码

**可以让PsyGo：**

- 工作的时候随时把代码推送到Gitee上（需要你给token）
- 在他自己的电脑上开发完整的Web App，然后直接部署上线
- 出PDF调研报告然后发过来
- 把代码打成压缩包发过来（但相比之下，更建议使用git）

## 先把PsyGo当个全能的人，再探索能力边界

**默认把PsyGo当一个全能的人，而不是一个工具。**

PsyGo没有写死一套固定的工具链，而是支持无限上下文和通用工具使用能力。
理论上，PsyGo可以：

- 安装和使用任何软件，配置任何开发环境
- 使用任何编程框架
- 接入任何第三方API服务，包括调用AI大模型
- 在权限（token）充分的情况下直接将App应用部署上线

当然，由于模型能力限制，PsyGo现实中可能出现某些工具链用不好的情况。
**可以当PsyGo显示出力不从心时再加以引导，像带实习生一样，
而不是像用工具一样，一开始就限制死。**

**不要：**

- 把一个完整的任务自己拆成很多小步骤，让PsyGo一个一个做（不要像用Claude Code一样用PsyGo）
- 给PsyGo过于细致具体的指令，一步一步教他/她怎么做（比如限制特定技术栈）

**可以：**

- 给一些大致的指引，然后直接丢给PsyGo一个复杂的任务
- 给PsyGo更多的自由，让他/她自己决定具体怎么执行
