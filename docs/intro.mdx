# Psygo用户指南

## Psygo不是工具，不要在App上等结果

Psygo无法在你发消息后立刻回复，因为Psygo不是聊天机器人，而是能干活的工人。

正确的交互方式是：**下命令之后，就关掉客户端窗口，然后该干啥干啥**。
Psygo把工作做完了，或者需要你提供消息了，自然会来找你。
就像一个人一样。

可以把Psygo当作微信上的一个联系人。正在干活的人是很难秒回的，但是他/她干完了自然会联系你。

**Psygo的设计理念就不是让你一直呆在App上**，
而是反过来，让AI尽量自主工作，尽可能省心，把更多的时间还给用户。

## 不要把Psygo App当主要交互方式

**Psygo不是聊天机器人，也不是AI工具，而是AI工人。**

**不要用ChatGPT/Claude Code的方式使用Psygo。**

**Psygo App只是管理员后台+微信，不是Psygo的全部**，
就像一个人远不只是他/她的微信账号。

实际的交付应该是通过发文件，git等等，而不是在Psygo App上发消息，
**就像没有人会把代码文件复制粘贴到微信里**。

**不要让Psygo：**

- 搞复杂的调研，然后在聊天框里汇报
- 在你的电脑上操作，比如给你本地配环境，在你本地写代码
- 像Claude Code一样在你本地调代码

**可以让Psygo：**

- 工作的时候随时把代码推送到Gitee上（需要你给token）
- 在他自己的电脑上开发完整的Web App，然后直接部署上线
- 出PDF调研报告然后发过来
- 把代码打成压缩包发过来（但相比之下，更建议使用git）

## 先把Psygo当个全能的人，再探索能力边界

**默认把Psygo当一个全能的人，而不是一个工具。**

Psygo没有写死一套固定的工具链，而是支持无限上下文和通用工具使用能力。
理论上，Psygo可以：

- 安装和使用任何软件，配置任何开发环境
- 使用任何编程框架
- 接入任何第三方API服务，包括调用AI大模型
- 在权限（token）充分的情况下直接将App应用部署上线

当然，由于模型能力限制，Psygo现实中可能出现某些工具链用不好的情况。
**可以当Psygo显示出力不从心时再加以引导，像带实习生一样，
而不是像用工具一样，一开始就限制死。**

**不要：**

- 把一个完整的任务自己拆成很多小步骤，让Psygo一个一个做（不要像用Claude Code一样用Psygo）
- 给Psygo过于细致具体的指令，一步一步教他/她怎么做（比如限制特定技术栈）

**可以：**

- 直接丢给Psygo一个复杂的任务
- 给Psygo更多的自由，让他/她自己决定具体怎么执行
